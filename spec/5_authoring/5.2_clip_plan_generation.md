# 5.2 Clip Plan Generation

## Goal

Create a shot-level clip planning artifact that turns scene intent into a renderable unit.

## Working Interpretation

- `clip = cut` in the current implementation.
- A clip plan is the bridge between scene analysis and prompt writing.
- The clip plan is allowed to use stable IDs and production language.
- The clip plan is not itself a prompt package.
- The initial chapter-based authoring pilot stays file-first before SQLite migration.

## Responsibilities

- Name the clip or shot
- State the beat or shot purpose
- Break the shot into textual 3-5 second motion segments
- Decide what kind of shot-start problem this clip is solving
- Decide whether the opening keyframe can be generated before prior cut videos finish
- Identify whether the shot opens from text and shared refs, a prior approved keyframe, or a prior approved video last frame
- Decide whether motion should preserve the approved keyframe's lighting and color grade or introduce an explicit planned look transition
- Identify which visible characters should be checked against approved character-sheet refs
- Decide whether optional automated consistency assist is appropriate for this clip
- Identify required refs
- Identify expected outputs
- Identify continuity changes between the approved keyframe, optional still-fix, and later video handoff

## Required Planning Details

- Every clip plan must say what the first visible frame of the cut should be.
- Every clip plan must describe what changes over time after that opening frame.
- The motion plan should describe subjects, camera, and environment separately when possible.
- Duration belongs in plan metadata, not later prompt body text.
- Clip planning starts only after scene decomposition and beat-bundle extraction are complete for that scene.
- A normal movie-style cut should plan as one short image-to-video segment by default.
- Longer clips, such as around 10 seconds, may declare multiple sequential motion segments instead of forcing one extended workflow by default.
- A normal approximately 5-second cut should not require chained end-frame-to-next-frame generation.
- A normal short cut should preserve the approved keyframe's lighting direction, color temperature, and overall grade unless the clip plan explicitly calls for a visible lighting change.
- Most clips should default to one of:
  - `reframe_same_moment`
  - `reblock_same_scene`
  - `insert`
  - `cutaway`
- `continuous_follow` should be rare and should be chosen only when the next cut genuinely depends on exact carry-forward from the previous motion result.
- For most clips, the default starting-keyframe strategy should not require the previous shot video to finish.
- The LLM should choose whether the opening keyframe should come from:
  - scene refs and beat staging alone
  - a prior approved keyframe as a reframing source
  - the previous approved video last frame as a camera-repositioning source
  - or, rarely, direct previous-last-frame continuity
- The default escalation path for a failed next-shot keyframe should be:
  - try the planned opening-keyframe strategy first
  - if review says the candidates are not good enough, escalate to `previous_video_last_frame_to_reframe`
  - do not make direct last-frame carry-forward the universal first choice
- `independent` and `soft_ref_previous` clips should be eligible for parallel keyframe generation before earlier cut videos complete.
- If a clip continues from earlier approved material, the plan must say whether continuity comes from:
  - the approved keyframe
  - the approved still-fix
  - the previous approved video last frame
- If a clip contains visible hero characters, the plan should say whether an optional identity-consistency assist is worth attempting after keyframe generation.
- Character-to-scene mapping is a prerequisite for useful automated still-fix decisions, because the system must know which approved character refs are actually visible in the clip before it can request corrective identity work.
- The preferred automatic identity-consistency method is conservative reference-guided image-to-image using approved character-sheet refs, not a full destructive redraw.
- Multi-character shots should prefer region-aware reference conditioning when available.
- If region-aware targeting is unavailable, the plan should either:
  - use a very conservative dominant-character consistency pass
  - or skip the automatic assist and stay on the normal review path
- Manual inpaint remains the escalation path when automated targeting cannot isolate the right subject cleanly.
- Anatomy repair should be planned separately from identity consistency.
- The preferred order is:
  - get identity and composition right in the approved keyframe
  - optionally run still-image anatomy repair
  - avoid relying on video-only hand repair as the default correction path
- If the clip intentionally changes the look during motion, that change should be planned explicitly rather than left for the video model to improvise.

## Shot-Start Decision Matrix

- `reframe_same_moment`
  - same dramatic beat, new camera angle, lens, or framing
  - usually uses `scene_refs_to_keyframe` or `approved_keyframe_to_reframe`
  - usually `independent` or `soft_ref_previous`
- `reblock_same_scene`
  - same room and scene family, but character placement or blocking changes
  - usually uses `scene_refs_to_keyframe`
  - usually `independent`
- `insert`
  - detail shot of a prop, hand, object, screen, or gesture
  - usually uses `scene_refs_to_keyframe`
  - usually `independent`
- `cutaway`
  - supplementary shot away from the main action
  - usually uses `scene_refs_to_keyframe`
  - usually `independent`
- `scene_reset`
  - new beat, time shift, or intentional composition reset
  - usually uses `scene_refs_to_keyframe`
  - usually `independent`
- `continuous_follow`
  - direct carry-forward from a previous motion result
  - should usually use `direct_previous_video_last_frame`
  - usually `hard_ref_previous`

## Optional Consistency Assist Modes

- `review_triggered_finalist_only`
  - recommended default
  - after keyframe review, run consistency assist only on the chosen primary or top finalists
  - best balance of compute cost and safety
- `post_batch_optional`
  - optional aggressive mode
  - run consistency assist on all generated keyframe candidates immediately after batch creation
  - useful for experiments, but more expensive and more likely to over-correct good compositions
- `off`
  - no automatic consistency assist
  - rely on normal keyframe review and later still-fix decisions

## Output Mapping

- Each clip plan should name the prompt files it expects to produce:
  - `03_prompt_packages/scenes/<scene_id>/<clip_id>/<scene_id>_<clip_id>_scene_stage_prompt.md`
  - `03_prompt_packages/keyframes/<scene_id>/<clip_id>/<scene_id>_<clip_id>_keyframe_prompt.md`
  - `03_prompt_packages/fixes/<scene_id>/<clip_id>/<scene_id>_<clip_id>_fix_01_prompt.md`
  - `03_prompt_packages/cut_motion/<scene_id>/<clip_id>/<scene_id>_<clip_id>_cut_motion_prompt.md`
- Each clip plan should name the expected render outputs:
  - approved keyframe
  - optional still-fix
  - cut-motion videos

## Planned Commands

- `python -m orchestrator plan-scene <project_slug> --scene <scene_id>`
  - creates or refreshes the clip roster and all clip plans for one scene
- `python -m orchestrator plan-clip <project_slug> --scene <scene_id> --clip <clip_id>`
  - regenerates one clip plan without touching the rest of the scene

## Handoff

- input:
  - scene analysis files
  - character and environment breakdowns
  - project continuity rules
- output:
  - one clip roster
  - one clip plan per clip
  - no prompt packages yet

## Acceptance

- Each clip has a single planning file that the runner, prompt writer, and reviewer can all reference.
- The planning file is detailed enough for the prompt writer to produce opener and continuation prompts without inventing continuity logic ad hoc.
- The planning file is detailed enough for the motion prompt writer to describe visible motion without restating every appearance detail from scratch.
- The planning file makes it explicit whether the clip should be satisfied by one short image-to-video generation or by multiple planned motion segments.
- The planning file makes it explicit whether a clip can generate its opening keyframe in parallel with other clips or must wait on a previous approved video result.
- The planning file makes the review fallback path explicit when the default opening-keyframe strategy is not good enough.
- The planning file makes it explicit whether optional identity-consistency or anatomy-repair automation should be attempted for the clip.
- The planning file makes it explicit whether motion should preserve the approved keyframe look or perform a planned lighting transition.
- For the first chapter-based validation, at least one scene should yield one or two prompt-ready clip plans before we introduce SQLite synchronization.
