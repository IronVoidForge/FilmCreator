# 5.2 Clip Plan Generation

## Goal

Create a shot-level clip planning artifact that turns scene intent into a renderable unit.

## Working Interpretation

- `clip = cut` in the current implementation.
- A clip plan is the bridge between scene analysis and prompt writing.
- The clip plan is allowed to use stable IDs and production language.
- The clip plan is not itself a prompt package.

## Responsibilities

- Name the clip or shot
- State the beat or shot purpose
- Break the shot into textual 3-5 second motion segments
- Identify whether the shot opens from text and shared refs or continues from an approved prior frame
- Identify required refs
- Identify expected outputs
- Identify continuity changes between the approved keyframe, optional still-fix, and later video handoff

## Planned Clip Plan Schema

- `Title`
- `ID`
- `Scene Purpose`
- `Clip Purpose`
- `Duration Seconds`
- `Start Mode`
  - `shared_refs_text`
  - `approved_keyframe`
  - `previous_approved_video_last_frame`
- `Shared Ref Targets`
- `Required Refs`
- `Optional Refs`
- `Opening Keyframe Intent`
- `Cut Motion Intent`
- `Interval Beats`
  - one bullet per 3-5 second motion segment
- `Continuity Rules`
- `Expected Prompt Files`
- `Expected Outputs`
- `Sources`

## Required Planning Details

- Every clip plan must say what the first visible frame of the cut should be.
- Every clip plan must describe what changes over time after that opening frame.
- The motion plan should describe subjects, camera, and environment separately when possible.
- Duration belongs in plan metadata, not later prompt body text.
- A normal movie-style cut should plan as one short image-to-video segment by default.
- Longer clips, such as around 10 seconds, may declare multiple sequential motion segments instead of forcing one extended workflow by default.
- A normal approximately 5-second cut should not require chained end-frame-to-next-frame generation.
- If a clip continues from earlier approved material, the plan must say whether continuity comes from:
  - the approved keyframe
  - the approved still-fix
  - the previous approved video last frame

## Output Mapping

- Each clip plan should name the prompt files it expects to produce:
  - `03_prompt_packages/scenes/<scene_id>/<clip_id>/<scene_id>_<clip_id>_scene_stage_prompt.md`
  - `03_prompt_packages/keyframes/<scene_id>/<clip_id>/<scene_id>_<clip_id>_keyframe_prompt.md`
  - `03_prompt_packages/fixes/<scene_id>/<clip_id>/<scene_id>_<clip_id>_fix_01_prompt.md`
  - `03_prompt_packages/cut_motion/<scene_id>/<clip_id>/<scene_id>_<clip_id>_cut_motion_prompt.md`
- Each clip plan should name the expected render outputs:
  - approved keyframe
  - optional still-fix
  - cut-motion videos

## Planned Commands

- `python -m orchestrator plan-scene <project_slug> --scene <scene_id>`
  - creates or refreshes the clip roster and all clip plans for one scene
- `python -m orchestrator plan-clip <project_slug> --scene <scene_id> --clip <clip_id>`
  - regenerates one clip plan without touching the rest of the scene

## Handoff

- input:
  - scene analysis files
  - character and environment breakdowns
  - project continuity rules
- output:
  - one clip roster
  - one clip plan per clip
  - no prompt packages yet

## Acceptance

- Each clip has a single planning file that the runner, prompt writer, and reviewer can all reference.
- The planning file is detailed enough for the prompt writer to produce opener and continuation prompts without inventing continuity logic ad hoc.
- The planning file is detailed enough for the motion prompt writer to describe visible motion without restating every appearance detail from scratch.
- The planning file makes it explicit whether the clip should be satisfied by one short image-to-video generation or by multiple planned motion segments.
