# 5.1 Story Analysis Outputs

## Goal

Define the analysis files that authoring should produce before prompt writing.

## Local Runtime Contract

- Authoring talks to a local LM Studio instance through its OpenAI-compatible API.
- Rendering does not depend on LM Studio remaining open after authoring completes.
- Analysis output is file-first. The runner later consumes files, not live LLM responses.
- Every LM Studio invocation should perform one bounded authoring task only.
- Higher-level authoring commands may orchestrate multiple LM Studio calls, but they should not collapse chapter analysis, scene planning, clip planning, and prompt writing into one monolithic response.
- Chapter-analysis and scene-planning tasks should prefer tagged Markdown packets plus parser-side validation over strict nested JSON.

## Required Outputs

- Project story summary
- Chapter-level analysis summaries
- Scene breakdowns grouped by scene ID
- Beat bundles or staging packets grouped by scene ID and beat ID
- Shot lists or clip rosters grouped by scene ID
- Character breakdowns keyed by asset ID
- Environment breakdowns keyed by asset ID
- Clip plans grouped by scene ID and clip ID, each with a 3-5 second motion-segment breakdown and a starting-keyframe strategy

## Planned Files

- `02_story_analysis/story_summary/project_summary.md`
- `02_story_analysis/chapter_analysis/CH###_summary.md`
- `02_story_analysis/scene_breakdowns/<scene_id>.md`
- `02_story_analysis/beat_bundles/<scene_id>/BEAT_INDEX.md`
- `02_story_analysis/beat_bundles/<scene_id>/<beat_id>.md`
- `02_story_analysis/character_breakdowns/<asset_id>.md`
- `02_story_analysis/environment_breakdowns/<asset_id>.md`
- `02_story_analysis/clip_plans/<scene_id>/<scene_id>_clip_roster.md`
- `02_story_analysis/clip_plans/<scene_id>/<clip_id>.md`

## Required Analysis Content

- `project_summary.md`
  - overall premise
  - major recurring characters
  - major recurring environments
  - continuity-sensitive visual rules
- `chapter_analysis/CH###_summary.md`
  - chapter summary
  - chapter-local visual motifs
  - present and referenced character groups
  - chapter-local environment families
  - continuity-sensitive facts the later scene files must preserve
- `scene_breakdowns/<scene_id>.md`
  - scene purpose
  - emotional and dramatic function
  - ordered beats
  - likely coverage families inside each beat
  - likely dominant axis or screen-direction rules when relevant
  - likely shared refs needed for the scene
- `beat_bundles/<scene_id>/BEAT_INDEX.md`
  - ordered beat IDs
  - one-line beat purposes
  - notes about which beats are the best first clip-planning candidates
- `beat_bundles/<scene_id>/<beat_id>.md`
  - beat purpose
  - room geography and dominant axis
  - character placement and eyeline logic
  - prop state and environmental state
  - reusable staging facts that multiple clips can share
  - likely coverage families such as master, over-the-shoulder, single, insert, and cutaway
- `character_breakdowns/<asset_id>.md`
  - physical description
  - costume or silhouette rules
  - continuity-critical traits
  - useful descriptive noun phrases for later prompt writing
  - explicit manual-description-needed flag when the source chapter does not contain enough stable visual detail
- `environment_breakdowns/<asset_id>.md`
  - architectural or geographic description
  - lighting and atmosphere rules
  - stable environmental elements
  - useful descriptive noun phrases for later prompt writing
- `clip_roster`
  - ordered clip IDs for the scene
  - one-line purpose for each clip
  - target duration for each clip
  - beat ID for each clip
  - continuity mode for each clip
  - composition type for each clip
  - dependency policy for each clip
- `clip_plans/<scene_id>/<clip_id>.md`
  - shot or cut purpose
  - beat ID
  - opening state at cut start
  - textual 3-5 second motion-segment beats
  - continuity mode
  - composition type
  - starting-keyframe strategy
  - dependency policy
  - auto-advance policy
  - review fallback strategy
  - required and optional refs
  - expected prompt-package targets

## Rules

- Analysis files live under `02_story_analysis/`.
- The pre-SQL pilot remains file-first.
- Analysis first decomposes story scenes into shot-level clips.
- Each clip plan records the shot intent and the interval beats used by prompt writing.
- Analysis is descriptive input to prompt writing, not a replacement for prompt packages.
- Clip plans bridge scene intent into clip-local render units.
- Analysis may use proper nouns and asset IDs internally so humans can follow the files easily.
- Analysis should identify visual facts, motion intent, and continuity rules separately so prompt writing does not have to infer them ad hoc.
- If a character is underdescribed for dependable reference generation, analysis should raise a manual-description-required flag instead of hallucinating missing details.
- The file-first pre-SQL flow may create a manual paste target under `01_source/character_descriptions/<asset_id>_manual_description.md` when that flag is raised.
- Analysis should decide, for every clip, what kind of opening-keyframe problem it is solving:
  - `reframe_same_moment`
  - `reblock_same_scene`
  - `insert`
  - `cutaway`
  - `scene_reset`
  - `continuous_follow`
- Analysis should treat `continuous_follow` as the rare exception, not the default for scene coverage.
- Analysis should default most clips to opening-keyframe strategies that do not require the previous cut video to finish first.
- Analysis should explicitly decide whether a clip is:
  - `independent`
  - `soft_ref_previous`
  - `hard_ref_previous`
- Analysis should support a review-triggered fallback path where a failed next-shot keyframe can be regenerated by using the previous approved video last frame as an image-to-image camera-repositioning source.
- Story analysis should not mention ComfyUI node details, workflow JSON details, or render patch points.
- The first chapter-based authoring validation should aim to produce one scene with one or two prompt-ready clips before scaling to the full chapter.

## Planned Commands

- `python -m orchestrator lmstudio-check`
  - verifies LM Studio connectivity, configured model name, and local API reachability
- `python -m orchestrator analyze-scene <project_slug> --scene <scene_id>`
  - reads source material and writes the analysis files for one scene
- `python -m orchestrator analyze-project <project_slug>`
  - optional later convenience command that writes project summary plus all scene-level analysis

## Handoff

- input:
  - local source text under `01_source/`
  - project and scene IDs
  - LM Studio model configuration
- output:
  - analysis files under `02_story_analysis/`
  - no render jobs submitted
  - no prompt packages written yet

## Acceptance

- Prompt writing can trace shot prompts and interval prompts back to stable analysis files.
- One scene can be analyzed with LM Studio running and ComfyUI closed.
- Analysis output names and locations are stable enough that later prompt-writing commands can overwrite or refresh them predictably.
